# AGENTS.md

Operational guide for AI agents working in this repo. End users read `README.md`;
this file is the "how it actually runs + how to test it" reference.

> TL;DR: cross-platform AI chat **+ a safety-inspection app** on top of a
> SafetyCulture/"Beacon" Convex backend. Field technicians fill safety forms in the
> **native** app (manually **or** via the AI assistant); office staff view/manage
> them in the **web** app. Everything runs fully offline for local dev.

---

## 1. What this is

A Turborepo + pnpm monorepo:

| Path | What | Stack |
|---|---|---|
| `apps/native` | Technician mobile app | Expo SDK 56, Expo Router, uniwind, `@expo/ui` |
| `apps/web` | Office console + chat | Next.js 16 (App Router), Tailwind v4 / shadcn |
| `packages/backend` | Convex backend | schema, queries/mutations/actions, `http.ts` |

Shared: one Convex backend, AI SDK v5 (`/chat` HTTP action streams via the Vercel AI
Gateway), and a WorkOS + guest auth model.

### Safety-inspection domain (the "Beacon" backend)

`templates → templateVersions (sections[].questions[]) → inspections (responses[])`.
~23 tables + Convex components (`@convex-dev/workflow`, `@convex-dev/aggregate`,
`@convex-dev/action-retrier`). Question types: `passFailNA`/`question`, `text`,
`number`, `temperature`, `multipleChoice`, `checkbox`, `date`, `signature`, `photo`,
`slider`, `list`, `instruction`.

**Demo data** (`convex/seed.ts`, run `pnpm --filter @packages/backend seed`):
- Org slug **`northwind`** (users: Josh Miller / Maria Lopez), 3 sites, 131 templates.
- Three featured demo forms surfaced first everywhere:
  **Daily Site Safety Walk**, **Forklift Pre-Start Check**, **Working at Heights Permit**.

### Key backend functions (`packages/backend/convex/`)

- `me.current` (query) / `me.ensureUser` (mutation) — bridge WorkOS/guest identity to a
  Beacon org+user in the `northwind` demo org. Apps call `ensureUser` once after sign-in.
- `templates.list {orgId}`, `templates.getWithVersion {templateId, version?}`.
- `inspections.start {orgId, templateId, inspectorId, siteId?, assetId?}` → inspectionId.
- `inspections.get {inspectionId}` → `{inspection, templateName, sections, scoringEnabled}`.
- `inspections.setAnswer {inspectionId, questionId, value?, note?, flagged?}` — one answer.
- `inspections.saveResponses {inspectionId, responses}` — bulk save (draft).
- `inspections.complete {inspectionId}` → `{score, flaggedQuestionIds, actionsCreated, workflowId}`
  (computes score, auto-creates corrective actions, runs the post-complete workflow).
- `inspections.list {orgId, status?}` → rows enriched with `templateName` + `inspectorName`.

### AI form-fill (`http.ts` `POST /chat`)

`/chat` accepts `{ messages, inspectionId? }`. When `inspectionId` is present it enters
**inspection-assist mode**:
- With `AI_GATEWAY_API_KEY` set → real model via `streamText` + tools
  (`getInspectionForm`, `setAnswer`, `completeInspection`), `stopWhen: stepCountIs(12)`.
- Without a key (default offline) → a **scripted agent** fills + completes the inspection
  using the real Convex mutations while streaming the same tool-call chunks.

Both apps render those tool chunks as cards (waiting / running / done / error). The
native runner's "✨ Complete with the assistant" button and the web
`/office/inspections/[id]` "Complete with the assistant" link both deep-link to the
chat with `inspectionId`.

## 2. Running locally

All three commands run from the repo root unless noted. **Local Convex is anonymous**
(`CONVEX_AGENT_MODE=anonymous`). Ports default to `3210` (API/WS) / `3211` (HTTP/site),
but a Convex backend already running for another project bumps you to `3212/3213`, etc.

```bash
# 1. Backend (persistent — keep it running; serves 3210/3211 or next free pair)
cd packages/backend && CONVEX_AGENT_MODE=anonymous npx convex dev --tail-logs disable
#    then, once: seed the demo org + 131 templates
pnpm --filter @packages/backend seed          # convex run seed:seedAll

# 2. Web
pnpm --filter web dev                          # http://localhost:3000  (chat) + /office

# 3. Native (custom dev build — NOT Expo Go)
pnpm --filter native ios                       # expo run:ios  (builds + installs + Metro)
```

Env files are **git-ignored** (`apps/*/.env.local`): they point the apps at the Convex
deployment. The committed default is `3210/3211`. If your backend landed on a different
pair, set `EXPO_PUBLIC_CONVEX_URL` / `EXPO_PUBLIC_CONVEX_SITE_URL` (native) and
`NEXT_PUBLIC_CONVEX_URL` / `NEXT_PUBLIC_CONVEX_SITE_URL` (web) to match.

**Bundle id / scheme:** `com.example.chat`, scheme `chat` (from `app.json`; `ios/` is
git-ignored and regenerated by `expo prebuild`).

### Gates before committing (run on changed files)

```bash
pnpm --filter @packages/backend typecheck      # tsc -p convex
pnpm --filter native typecheck && pnpm --filter web typecheck
npx @biomejs/biome check <changed-files>       # linter (autofix with --write)
```

## 3. E2E device testing with Argent (Haiku-optimized)

This is the fast, repeatable way to prove the three forms work on device each time.
**Model tiering:** dispatch the on-device driving to a **Haiku** agent — it is
*observe-and-report only* and must **never edit code**. Setup/build/debugging that
needs reasoning stays with Sonnet/Opus.

**Preflight (run first, every time):** verifies backend + seed + Metro + lists sims in
one shot — observe-only, safe to run repeatedly.
```bash
scripts/e2e-mobile-preflight.sh                      # defaults to API:3210 / site:3211
API_PORT=3212 SITE_PORT=3213 scripts/e2e-mobile-preflight.sh   # if your backend bumped ports
```
It prints `READY ✓` (or the failing checks) and exports the demo `ORG` id for §3.3.

### 3.0 One-time-per-run setup (do this on the main/Sonnet thread, not Haiku)

1. **Use a DEDICATED clean simulator.** Do not reuse a simulator another project is
   driving — app name/scheme collisions make the expo-dev-client load the *other*
   project's bundle (you'll see a different app + an `AsyncStorage is null` crash loop).
   ```bash
   xcrun simctl list devices | grep -i booted          # see what's already in use
   xcrun simctl boot <UDID-of-an-idle-device>          # boot a clean one
   open -a Simulator
   ```
2. **Backend up + seeded** (§2). Confirm: `curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:<sitePort>/guest-login -X POST -H 'Content-Type: application/json' -d '{"subject":"t"}'` → `200`.
3. **Build + install on that clean sim** (also starts Metro on the chosen port):
   ```bash
   cd apps/native
   RCT_METRO_PORT=8082 npx expo run:ios --device "<UDID>" --port 8082
   ```
   If a stale build references an old path (`virtual filesystem overlay ... React-VFS.yaml
   not found`) after a repo rename/move: `rm -rf ios/build ~/Library/Developer/Xcode/DerivedData/chat-* && (cd ios && pod install)`, then rebuild.

### 3.1 Argent gotchas (bake these into the Haiku prompt)

- **Discovery before every tap** (never derive coords from a screenshot). Primary tool:
  `native-describe-screen --udid <UDID> --bundleId com.example.chat` — returns
  `screenFrame {width,height}` + `elements[].tapPoint {x,y}` in **points**; normalize:
  `nx = tapPoint.x / width`, `ny = tapPoint.y / height`.
- `describe` intermittently fails with `Unexpected end of JSON input` — fall back to
  `native-describe-screen` (needs `--bundleId`).
- **The first list row sits under the large-title nav bar** and is not tappable at its
  reported center. Tap list rows whose center `ny ≥ 0.13`, or tap a row's lower edge.
  Same for the runner's header "✨ Complete with the assistant" — tap its **lower edge**
  (`ny ≈ 0.128`), not its center.
- **Keyboard typing works as a standalone `keyboard` call but FAILS inside a
  `run-sequence` batch** (timing). Type as separate calls: tap field → `keyboard --text` → tap send.
  Pure taps (Pass/Fail buttons) batch fine in `run-sequence`.
- **Don't navigate away from the runner before Complete** — answers live in local state
  until Save draft / Complete; leaving loses them.
- Coords are normalized `[0,1]`. `gesture-tap --x --y`; `gesture-swipe --x --y --to-x --to-y`.

### 3.2 The flow (per form)

**Manual fill** (best on a Pass/Fail-heavy form, e.g. Working at Heights / Forklift):
1. Login screen → tap **Continue as guest** (fresh sim → no session → login shows).
2. Drawer (top-left "List") → **Inspections** → tap a demo form row (`ny ≥ 0.13`).
3. Runner renders sections. Batch-tap "Pass" on each `passFailNA` question
   (`x ≈ 0.107`); scroll; repeat. Skip `signature`/`photo` (placeholders in demo).
4. Tap **Complete** → expect alert **"Inspection complete · Score: NN% · Corrective actions: N"**.

**AI fill** (conversational):
1. From the runner, tap **"✨ Complete with the assistant"** (lower edge, `ny ≈ 0.128`).
2. Chat opens bound to that `inspectionId`. Tap the input → `keyboard --text "Please fill
   out and complete this inspection."` → tap send (up-arrow, `x ≈ 0.925`).
3. Expect streaming tool-call cards: `getInspectionForm` → `setAnswer` (×N, "Done") →
   `completeInspection` → assistant summary "Done — answered N item(s), scored NN%…".

### 3.3 Verify in the backend (ground truth, not just pixels)

```bash
cd packages/backend
ORG=$(CONVEX_AGENT_MODE=anonymous npx convex data organizations 2>/dev/null \
      | python3 -c "import sys,re;print(re.search(r'\"_id\":\s*\"([a-z0-9]+)\".*northwind',sys.stdin.read(),re.S).group(1))")
CONVEX_AGENT_MODE=anonymous npx convex run inspections:list "{\"orgId\":\"$ORG\",\"status\":\"completed\"}" \
  | python3 -c "import sys,json;a=json.loads(sys.stdin.read());[print(i['templateName'],'->',i['status'],i['score']) for i in a[:6]]"
# expect each demo form -> completed 100
```

### 3.4 Ready-to-dispatch Haiku agent prompt

> You are a QA agent driving an iOS simulator via the Argent CLI (`argent run <tool>`).
> **Observe and report only — never edit any file.** Target UDID: `<UDID>`,
> bundle id `com.example.chat`.
>
> Before every tap, run `argent run native-describe-screen --udid <UDID> --bundleId
> com.example.chat`, find the element by label, and tap its `tapPoint` normalized by
> `screenFrame` (nx=x/width, ny=y/height). If `describe` errors, use
> `native-describe-screen`. Tap list rows only when `ny ≥ 0.13`; tap the
> "✨ Complete with the assistant" button at its lower edge. Type with standalone
> `keyboard --text` calls (never inside `run-sequence`). Take a `screenshot --out`
> before/after each milestone.
>
> Steps: (1) If the login screen shows, tap "Continue as guest". (2) Open the drawer →
> Inspections. (3) Manually fill **Working at Heights Permit** (tap "Pass" on every
> Pass/Fail/N/A question, scrolling as needed) → tap **Complete** → screenshot the
> "Inspection complete" alert and report the score. (4) Back to Inspections → open
> **Forklift Pre-Start Check** → tap "✨ Complete with the assistant" → send "Please
> fill out and complete this inspection." → wait ~6s → screenshot the tool-call cards
> and the final "scored NN%" summary. (5) Report, per form: did it reach `completed`
> and what score. Attach screenshot paths. Do not fix anything; just report failures.

> For a repeatable sequence, record it once with the `argent-create-flow` skill and
> replay it on later runs instead of re-deriving every tap.

### 3.5 Cleanup when done

```bash
argent run stop-all-simulator-servers
# stop Metro (note your port) and the local Convex backend if you started them
```

## 4. Known cross-project hazards on this machine

- A separate project (`taskr-convex-test`) frequently runs its own Convex backend
  (`3210/3211`) and Metro (`8081`), and ships an app that **shares our `chat` scheme**.
  Symptoms when you collide: your dev-client loads *their* bundle / a "Taskr" login /
  an `AsyncStorage is null` crash loop. Mitigation: dedicated clean sim + a unique
  Metro port (`8082`) + point `.env.local` at *your* Convex ports.
- `expo start` does **not** stay alive headlessly in a background shell; use
  `expo run:ios` (keeps Metro alive) or keep a foreground terminal for Metro.
